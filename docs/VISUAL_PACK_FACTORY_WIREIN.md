## 2. Lifecycle

```
sources/                      candidates/                   exports/approved/      source/images/<asset>.png
  add_source       →   ingest (generate_pack)  →   queue_review  →
                                                              approve_candidate (final)
                                                              →  hardware_review (device photo)
                                                              →  export_candidate
                                                              →  update_visual_spec
```

States: `generated → queued_for_review → approved_for_iteration ↘`
`→ rejected / needs_correction ↘ → approved_final → hardware_reviewed → exported`

**Two rules cannot be bypassed:**
1. `generated → approved_final` is forbidden without `--reviewer` (human signoff recorded in `approvals.yaml`).
2. After v0.2.0, `validate_pack --enforce-hardware` flags any `approved_final` lacking a `hardware_review` record. Simulator screenshots do NOT count.

## 3. What you must do in `hakcd-v4` (the wire-in)

### 3.1 V3 validator stays put. Factory emits into V3's schema.

The factory and the HAKCD V3 validator (`tools/canon/validate_visuals.sh`)
are TWO validators with TWO scopes — they coexist:

| Validator | Scope |
|---|---|
| HAKCD `tools/canon/validate_visuals.sh` (V3) | Build gate. Reads `source/data/visual_spec.lua` (44+ assets: images, audio, debug fixtures, tilesets). Enforces art_status discipline, path resolution, reviewer signoff, readability floor. WARN mode below pdxinfo 0.2.0, FAIL past gate. |
| Factory `services/visual_pack_factory/tools/validate_pack.py` | Workflow gate. Reads per-pack state (`pack_config.yaml`, candidate JSONs, approvals.yaml). Enforces state machine, dimension match, 1-bit color, source `usage=inspiration_only`, hardware-review presence. Returns exit `0`/`2`/`3`. |

Do **not** replace V3 with the factory wrapper. V3 owns assets the factory
doesn't manage (sfx, music, debug fixtures, hand-authored finals). The
factory only manages assets that come out of packs.

Factory `tools/update_visual_spec.py` is the bridge: it walks every pack,
takes candidates in `approved_final`/`exported` state, and emits each one as
a V3-schema entry into `<repo>/source/data/visual_spec.lua`. Existing
entries without a `source_pack` marker are preserved verbatim across
regenerations. An id collision between a factory candidate and a
hand-authored entry is a hard error unless
`--allow-overwrite-unmanaged` is passed.

The factory's own validator can still be invoked from the HAKCD build chain
if/when you want pack-state checks to gate the build — call it in addition
to V3, not in place of it.

### 3.2 `source/data/visual_spec.lua` schema (V3, what factory emits)

The factory merges its candidates into the V3 schema HAKCD's validator
already consumes. One pack-managed entry looks like this:

```lua
local visual_spec = {
    newb_character_pack_v003_b = {
        art_status = "final",
        type = "image",
        path = "images/newb-table-48-48",  -- relative, NO extension
        sheet_dimensions = { w = 48, h = 48 },
        target_dimensions = { w = 48, h = 48 },
        human_reviewed = true,
        reviewer = "cory",
        reviewed_at = "2026-05-26T12:00:00+00:00",
        meets_readability_min = true,
        readability_min_pct_screen = 0,
        reference_image = nil,
        target_replacement_version = nil,
        frame_count = 1,
        notes = "Authored via visual_pack_factory pack 'newb_character_pack'.",
        id = "newb_character_pack_v003_b",
        -- factory ownership markers (V3 ignores unknown fields):
        source_pack = "newb_character_pack",
        approved_candidate = "newb_character_pack_v003_b",
        hardware_reviewed = true,
        exported_to = "/home/hakcer/projects/hakcd-v4/source/images/newb-table-48-48.png",
    },
    -- ... other factory entries, plus hand-authored audio / debug / final entries preserved verbatim
}

function visual_spec.placeholders() ... end
function visual_spec.failing_readability() ... end
_G.visual_spec = visual_spec
return visual_spec
```

**Don't hand-edit factory-owned entries** (those with `source_pack`). The
factory regenerates them on every export. Hand-authored entries (audio,
debug fixtures, manually finalized art) are preserved across regenerations
and can be edited safely.

### 3.4 Convert existing real reference material into sources

You already curate references in `docs/concepts/`, `docs/reference/`,
`tools/sprites/`, etc. For each real reference relevant to a pack:

```bash
curl -X POST .../packs/<pack_id>/sources \
  -F "file=@/path/to/reference.png" \
  -F "type=image" \
  -F "notes=strong silhouette / hands legible"
```

URL refs and inline notes use `type=url` / `type=note` (no file). Every
source is tagged `usage=inspiration_only` automatically — the factory
forbids any other value. **Do not push copyrighted assets in as `final`
art.**

ONLY this call sets `hardware_reviewed=true`. Simulator screenshots are

## 4. REST surface (all auth-protected via existing 23studios middleware)

Base: `http://127.0.0.1:8090/api/projects/hakcd/visual-pack`
(or via the code-server proxy at `/proxy/8090/...`)

```
GET    /packs

POST   /packs/:packId/sources                          (multipart: file=… type=…)
GET    /packs/:packId/candidates
POST   /packs/:packId/candidates                       (multipart: file=…)
POST   /packs/:packId/candidates/:cid/approve          { level: "iteration"|"final", reviewer, notes }
POST   /packs/:packId/candidates/:cid/reject           { reason, reviewer, correction: bool, correction_md_path }
POST   /packs/:packId/candidates/:cid/export           { asset_name?, enforce_hardware: bool }
POST   /packs/:packId/candidates/:cid/hardware-review  (multipart: photo=…)

POST   /packs/:packId/contact-sheet                    { mode, tile_size, cols }
```

CSRF + auth are inherited from the global `/api` chain. The factory
process spawns `python3` (or the `.venv-visual-pack/bin/python` if it
exists) with `shell: false`. `services/visual_pack_factory/installer.sh`
sets the venv up; run it once on the server.

### Init pack example

```bash
curl -X POST .../packs \
  -H "Content-Type: application/json" \
  -H "x-csrf-token: $CSRF" \
  --cookie "$COOKIES" \
  -d '{
    "pack_id": "newb_character_pack",
    "type": "character_pack",
    "target_dim": "48x48",
    "description": "Newb protagonist sprite replacement (V4).",
    "export_target": "source/images"
  }'
```

### Approve final + export example

```bash
curl -X POST .../packs/newb_character_pack/candidates/newb_character_v003_b/approve \

curl -X POST .../packs/newb_character_pack/candidates/newb_character_v003_b/export \
  -H "Content-Type: application/json" \

# Then regenerate visual_spec.lua
curl -X POST .../visual-pack/spec \
```

## 5. Boundaries (do NOT do these things)

- **Don't** copy the factory into `hakcd-v4`. It lives in 23studios so every
  game can use it. Wire in by HTTP + the CLI wrapper, not by vendoring.
- **Don't** hand-edit `source/data/visual_spec.lua`. The factory rewrites it.
- **Don't** approve `final` without `--reviewer`. The API rejects.
- **Don't** mark an asset hardware-reviewed from a simulator screenshot.
  Only the `hardware-review` endpoint with a real device photo flips that
  flag.
- **Don't** export a candidate whose dimensions don't match
  `pack_config.target_dimensions`. The exporter refuses.
- **Don't** start writing your own placeholder detection in HAKCD. The
  ingest endpoint already runs `playdate_validator.isPlaceholderScenePng`
  before accepting any candidate, and rejects with HTTP 422
  `placeholder_rejected`.

## 6. Acceptance for V4–V8

Each phase is "done" when:

1. its pack exists (`pack_config.yaml` present)
2. its `style/style_guide.md` reflects V1's output
3. it has ≥ 3 final-approved candidates with `human_reviewed=true`
4. every final candidate has a hardware review record (`verdict: pass`)
5. `source/images/<asset>.png` files exist for every approved candidate
6. `source/data/visual_spec.lua` contains a row per asset
7. `tools/canon/validate_visuals.sh` exits 0 with `ENFORCE_HW=1`

When all five packs (V4–V8) hit those criteria, V10's visual signoff gate
can be flipped to ENFORCED in CI and `make build` will block any future
regression to placeholders.
